## Architecture Decisions
1. **Seed Fallback Pattern**: The new-construction API returns hardcoded seed projects when Supabase table is empty — ensures the UI always has data
2. **Service Role for APIs**: All dashboard API routes use service_role key (bypasses RLS) — RLS policies exist as defense-in-depth
3. **Share Tokens**: Collections generate `crypto.randomUUID()` share tokens for future public sharing
4. **Scraper Dedup**: Projects are deduplicated by `slug` during upsert using Supabase `resolution=merge-duplicates`
